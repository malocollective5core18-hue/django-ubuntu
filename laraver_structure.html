Production Server Baseline

Step: Prepare Production Host

Commands:

sudo apt update
sudo apt install php php-fpm php-cli php-mbstring php-xml php-curl php-zip unzip curl composer nodejs npm nginx redis-server -y

Description: Prepare the host with PHP-FPM, Composer, Node.js, Nginx, Redis, and the required PHP extensions.

Why needed: Production Laravel needs a stable runtime, web server, cache/queue backend, and locked system dependencies before releases are deployed.

  • Confirm PHP version, extensions, time zone, memory limits, and upload limits.
  • Create a deploy user instead of deploying as root.
  • Keep storage and bootstrap/cache writable by the web server user.
Release Source

Step: Pull Approved Release

Commands:

Clone release repository
cd release

Description: Deploy from a reviewed branch, tag, or CI artifact instead of editing files directly on the server.

Why needed: Production releases must be traceable, repeatable, and easy to roll back.

  • Deploy a Git tag, release branch, or CI artifact with a known commit SHA.
  • Keep a releases directory and a current symlink for clean switching.
  • Never hot-edit production files outside the release process.
Optimized PHP Dependencies

Step: Install Runtime Packages

Command:

composer install --no-dev --optimize-autoloader

Description: Installs only production Composer packages and builds the optimized autoloader.

Why needed: Removes development packages from the server and improves class loading performance.

Note: Run from CI or the release directory with a committed composer.lock file.

  • Use composer.lock so production installs exactly what was tested.
  • Fail the release if Composer install cannot complete cleanly.
  • Do not run composer update on the production server.
Build Frontend Assets

Step: Compile Production Assets

Command:

npm ci
npm run build

Description: Installs exact Node dependencies from package-lock.json and compiles Vite assets.

Why needed: Production should serve compiled static assets, not run the development Vite server.

Note: Build in CI when possible, then ship the generated public/build assets with the release.

  • Commit package-lock.json for deterministic npm ci builds.
  • Verify public/build exists before switching the release live.
  • Serve compiled assets with long-lived cache headers when safe.
Secrets & Environment

Step: Configure Production Secrets

Command:

Set secrets and APP_KEY

Description: Set APP_ENV=production, APP_DEBUG=false, APP_KEY, database credentials, cache, queue, mail, and third-party keys through secure server or CI secret storage.

Why needed: Secrets must not be committed to the repo, and production must never expose debug output.

Note: Generate APP_KEY once for the application and preserve it across releases.

  • Set APP_URL, trusted proxy settings, session domain, mail, cache, queue, and filesystem values.
  • Store .env outside the release directory and symlink it into each release.
  • Keep APP_DEBUG=false before any public traffic reaches the app.
Cache Framework Config

Step: Optimize Laravel Boot

Command:

php artisan config:cache
php artisan route:cache
php artisan view:cache
php artisan event:cache

Description: Caches configuration, routes, views, and events for faster production boot time.

Why needed: Laravel should avoid repeatedly parsing configuration and routes on every request.

Note: Rebuild caches after setting the production .env and before switching the release live.

Quality Gate

Step: Run Tests and Checks

Commands:

php artisan test
vendor/bin/pint --test
vendor/bin/phpstan analyse

Description: Run automated tests, formatting checks, and static analysis before deployment.

Why needed: A production release should fail before it reaches users, not after.

Note: Put this in CI so every release uses the same checks.

  • Run tests against a clean database or disposable test environment.
  • Block deployment when tests, linting, or static analysis fail.
  • Add smoke tests for login, important pages, and critical API routes.
Database Migrations

Step: Apply Database Migrations

Command:

php artisan migrate --force

Description: Applies reviewed database migrations without an interactive prompt.

Why needed: Production deployments must update schema reliably as part of the release.

Note: Back up the database first and design migrations to be safe for live traffic.

  • Take a database backup before destructive or high-risk migrations.
  • Prefer additive migrations for zero-downtime deploys.
  • Separate large data backfills from schema changes when possible.
Serve Traffic

Step: Run Behind Nginx and PHP-FPM

Command:

Nginx + PHP-FPM live

Description: Point Nginx to the Laravel public directory and pass PHP requests to PHP-FPM.

Why needed: The local Artisan web server is only for development; production needs a real web server and process manager.

Note: Enable HTTPS, compression, upload limits, security headers, and correct file permissions.

  • Nginx document root should be current/public, never the project root.
  • Only public assets should be web-accessible.
  • Reload Nginx and PHP-FPM after config or release changes.
Workers & Scheduler

Step: Run Background Work

Command:

php artisan queue:work
php artisan schedule:run

Description: Run queue workers with Supervisor or systemd and trigger the scheduler every minute from cron.

Why needed: Emails, jobs, notifications, and scheduled tasks must keep running outside web requests.

Note: Restart workers after each deploy with php artisan queue:restart.

  • Use Supervisor or systemd so queue workers restart after failure.
  • Add one cron entry that runs php artisan schedule:run every minute.
  • Monitor failed_jobs and alert when failures rise.
Observability

Step: Monitor Production Logs

Command:

Monitor logs

Description: Watch Laravel logs, web server logs, queue failures, application metrics, and error tracking after release.

Why needed: Production work is not finished until you know the release is healthy under real traffic.

Note: Use tools such as Laravel logs, queue:failed, system logs, uptime checks, and an error tracker.

  • Check storage/logs/laravel.log, Nginx logs, PHP-FPM logs, and worker logs.
  • Track response time, error rate, queue depth, database load, and disk usage.
  • Send exceptions to an external error tracker, not only local log files.
Release Planning

Step: Plan Release Branch

Command:

Plan release branch

Description: Choose the tag, branch, or build artifact that is approved for production.

Why needed: A release plan defines what ships, who approved it, what migrations run, and how rollback works.

  • Record release owner, commit SHA, migration risk, and rollback command.
  • Schedule high-risk releases for a low-traffic window.
  • Tell the team when deployment starts and when it is verified.
Rollback Plan

Step: Prepare Recovery Path

Command:

Rollback plan

Description: Keep the previous release, database backup, and rollback commands ready before switching traffic.

Why needed: If health checks fail, the team needs a fast, practiced way to return to the last stable version.

  • Keep at least one previous release directory available.
  • Know whether database migrations are reversible before deployment.
  • After rollback, restart workers and rebuild caches for the active release.
Health Checks

Step: Validate Live Application

Command:

Health checks

Description: Check homepage, login, critical APIs, database connectivity, queues, scheduler, and external services.

Why needed: Passing deployment commands is not the same as proving the system works for users.

Note: Automate these checks and run them immediately after switching the release live.

  • Verify HTTP 200 for public and authenticated critical paths.
  • Confirm queues process jobs and scheduler tasks are not failing.
  • Check logs for new exceptions during the first minutes after release.
Zero-Downtime Release

Step: Switch Current Symlink

Command:

Zero-downtime release

Description: Build each release in its own directory, share storage/.env, then atomically point current to the new release.

Why needed: Users should not see half-copied files or downtime while the application is being deployed.

  • Prepare the new release completely before changing the current symlink.
  • Use maintenance mode only when a migration cannot be done live.
  • Restart PHP-FPM/opcache and queue workers after the switch.
Release Directory

Step: Work Inside Release

Command:

cd release

Description: Run install, build, cache, and migration commands from the prepared release directory.

Why needed: Keeps the active production version untouched until the new release is fully ready.

Note: Shared directories usually include storage and the production .env file.

  • Symlink storage so uploaded files and logs survive releases.
  • Symlink .env from a protected shared location.
  • Run artisan commands from the release that is about to go live.
Repeatable Frontend Build

Step: Rebuild Assets If Needed

Command:

npm ci
npm run build

Description: Recreate frontend assets exactly from the lockfile when the release includes frontend changes.

Why needed: Production assets must be deterministic and match the backend code being released.

Note: Prefer CI-built assets for faster and cleaner server deployments.

  • Keep Node version consistent between CI and production builds.
  • Clear stale assets only after the new build is available.
  • Confirm the manifest file exists if the app uses Vite.

Production Readiness Checklist

  • CI passes tests, linting, static analysis, and asset build.
  • APP_ENV=production, APP_DEBUG=false, APP_KEY preserved, and secrets protected.
  • Database backup exists before migrations run.
  • Queues, scheduler, cache, storage links, and file permissions are configured.
  • Health checks, logs, metrics, alerts, and rollback path are ready before release.
